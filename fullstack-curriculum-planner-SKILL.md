# 풀스택 과정 기획 스킬

풀스택 개발 교육과정을 처음부터 설계하거나 기존 과정을 개선할 때 사용하는 스킬입니다.
개인 학습자의 진도 코칭이 아니라, **과정 자체를 만드는 기획자/강사** 관점에서 동작합니다.

## 사용 시점

- 부트캠프/사내교육/온라인 강의용 풀스택 커리큘럼을 새로 설계할 때
- 기존 커리큘럼의 차시 구성, 난이도, 시간 배분을 점검하거나 개선할 때
- 특정 대상(취준생, 신입 개발자, 비전공자 등)에 맞춘 커리큘럼 초안이 필요할 때
- 커리큘럼 문서(강의계획서, 차시표, 학습목표표)를 산출물로 만들어야 할 때

## 기획 프로세스

커리큘럼을 설계하기 전에 아래 항목을 먼저 확인하세요. 사용자가 이미 답을 줬다면 다시 묻지 말고
대화에서 추출해서 사용합니다. 누락된 항목만 짧게 되물으세요.

1. **교육 대상**: 비전공자 / 컴공 전공자 / 현직 개발자 전환 / 신입사원 온보딩 등
2. **과정 기간 및 시간**: 총 주차 수, 주당 시간(풀타임/파트타임)
3. **기술 스택 범위**: 프론트엔드/백엔드 프레임워크, DB, 배포 환경 등
4. **목표 수준**: 입문 → 실무 배치 가능 수준, 또는 포트폴리오 완성 수준 등
5. **평가 방식**: 프로젝트 기반 평가, 퀴즈, 코드리뷰, 최종 발표 등 포함 여부
6. **제약 조건**: 강사 수, 실습 환경(온라인/오프라인), 예산, 필수 포함 기술 등

## 산출물 구조

커리큘럼 기획 결과물은 아래 구조를 기본으로 작성합니다 (요청에 따라 표/문서 형태 조정 가능):

1. **과정 개요**: 대상, 기간, 목표, 선수 지식
2. **전체 로드맵**: 모듈(단계)별 제목 + 기간 + 핵심 키워드 한눈에 보는 표
3. **모듈별 상세**
   - 학습 목표 (이 모듈을 마치면 할 수 있는 것)
   - 세부 커리큘럼 항목
   - 실습/과제 (반드시 각 모듈에 최소 1개 실습 배정)
   - 예상 소요 시간
   - 평가 기준 (해당 시)
4. **최종 프로젝트/졸업 요건**: 통합 프로젝트 주제 예시, 필수 구현 요소, 평가 기준
5. **커리큘럼 검증 체크리스트** (아래 참고)

## 커리큘럼 설계 원칙

- **선수학습 순서 준수**: 각 모듈은 이전 모듈에서 배운 개념을 전제로 쌓이도록 배치
- **이론:실습 비율**: 실무 배치가 목표인 과정은 이론 30% : 실습 70% 수준을 권장
- **모듈당 최소 1개 실습 프로젝트**: 이론만으로 끝나는 모듈이 없도록 함
- **난이도 곡선 점검**: 특정 모듈에서 난이도가 급격히 뛰지 않는지 확인 (필요 시 보충 모듈 추가)
- **평가 가능성**: 각 모듈의 학습 목표는 "설명할 수 있다"보다 "구현할 수 있다/만들 수 있다"처럼
  관찰 가능한 동사로 서술
- **시간 현실성**: 주차별 학습 시간과 실습 난이도가 대상자 수준(비전공자 vs 현직자)에 맞는지 재확인

## 커리큘럼 검증 체크리스트

기획 결과물을 내놓기 전에 아래를 점검하세요.

- [ ] 각 모듈의 학습 목표가 명확하고 관찰 가능한 동사로 쓰였는가
- [ ] 모듈 순서가 기술적 선수 요건을 위반하지 않는가 (예: DB 연동 전에 API 기초를 배우는가)
- [ ] 모듈마다 실습/프로젝트가 배정되어 있는가
- [ ] 전체 기간 대비 각 모듈 시간 배분이 현실적인가
- [ ] 평가 방식이 학습 목표와 일치하는가
- [ ] 최종 프로젝트가 배운 전체 기술 스택을 통합적으로 사용하는가
- [ ] 대상자 수준(비전공자/현직자 등)에 맞는 난이도인가

## 참고

- 개인 학습자의 진도 코칭(내가 지금 어디까지 왔는지, 다음에 뭘 공부할지)이 필요한 경우는
  `fullstack-learning-coach` 스킬을 사용하고, 이 스킬과 혼동하지 않습니다.
- 커리큘럼 산출물을 정식 문서(워드/PDF)로 원할 경우 docx/pdf 스킬과 함께 사용합니다.
